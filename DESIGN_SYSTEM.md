# Daniel Akanmu Design Constitution

Version: 1.0

---

# Purpose

This document defines the design philosophy that every interface should follow.

The goal is not to create beautiful interfaces.

The goal is to create interfaces that feel inevitable.

Every visual element must exist for a reason.

Beauty is a consequence of clarity.

---

# Core Philosophy

Design removes friction.

Good design disappears.

Users should never think about the interface.

They should think only about what they are trying to accomplish.

If an element competes for attention without improving comprehension, remove it.

---

# Guiding Principle

Before adding anything ask:

> "What purpose does this serve?"

If the answer is unclear...

Do not add it.

---

# Visual Hierarchy

Hierarchy is the most important design tool.

Always establish:

1. Primary action
2. Supporting information
3. Secondary actions
4. Decorative elements

Nothing should compete with the primary purpose.

---

# Simplicity

Prefer fewer components.

Prefer fewer colors.

Prefer fewer fonts.

Prefer fewer interactions.

Complexity should exist in the system.

Never in the interface.

---

# Layout

Whitespace is not empty.

Whitespace guides attention.

Never compress content simply because more information can fit.

Give elements room to breathe.

---

# Alignment

Every element must align intentionally.

Never eyeball spacing.

Use consistent spacing systems.

Examples:

- 4
- 8
- 12
- 16
- 24
- 32
- 48
- 64

Maintain rhythm.

---

# Typography

Typography creates hierarchy.

Never rely on color alone.

Use:

- size
- weight
- spacing
- placement

before introducing additional colors.

Readable always wins.

---

# Color

Color communicates.

Never decorate.

Each color should have meaning.

Examples:

Primary

Accent

Warning

Success

Danger

Neutral

Avoid rainbow interfaces.

Muted palettes are preferred.

Let content provide richness.

---

# Contrast

Contrast creates focus.

Increase contrast only where attention is required.

Not everything deserves emphasis.

---

# Motion

Motion explains.

Never entertain.

Animations should answer:

Where did this come from?

Where did it go?

What changed?

If animation does not improve understanding...

Remove it.

---

# Components

Components should feel engineered.

Avoid decorative flourishes.

Borders should exist only when necessary.

Prefer elevation only when hierarchy requires it.

Rounded corners should be subtle.

Consistency matters more than originality.

---

# Icons

Icons support text.

Icons rarely replace text.

Never make users interpret symbols unnecessarily.

---

# Buttons

Buttons should communicate priority.

Primary action:

One.

Secondary:

Few.

Avoid multiple competing primary buttons.

---

# Forms

Reduce cognitive effort.

Ask only for necessary information.

Group related fields.

Always provide useful validation.

Never punish mistakes.

---

# Navigation

Navigation should disappear.

Users should know:

Where they are.

Where they can go.

How to return.

Without thinking.

---

# Content

Design starts with content.

Never create layouts before understanding the information.

Content determines structure.

Structure determines layout.

Layout determines visuals.

---

# Responsiveness

Design mobile first.

Expand naturally.

Never simply shrink desktop interfaces.

Every breakpoint should feel intentional.

---

# Accessibility

Accessibility is not optional.

Every interface should support:

Keyboard navigation.

Screen readers.

High contrast.

Readable typography.

Clear focus states.

Adequate touch targets.

---

# Performance

Performance is part of design.

Fast feels beautiful.

Avoid unnecessary libraries.

Avoid unnecessary rendering.

Avoid unnecessary images.

Avoid unnecessary animations.

---

# Empty States

Every empty state should answer:

What happened?

Why?

What should I do next?

Never leave users stranded.

---

# Errors

Errors should educate.

Never blame users.

Explain:

What happened.

Why.

How to fix it.

---

# Loading States

Never surprise users.

Communicate progress.

Skeletons are preferred over spinners.

---

# Microcopy

Every word matters.

Use simple language.

Prefer:

"Save"

Instead of

"Commit Changes"

Prefer:

"Delete"

Instead of

"Execute Removal"

Clarity beats cleverness.

---

# Design Process

Before designing:

Understand the problem.

Understand the users.

Understand the constraints.

Only then design.

---

# Decision Framework

When making design decisions ask:

Does this improve clarity?

Does this reduce effort?

Does this improve focus?

Does this improve trust?

Does this improve speed?

If not...

Reject it.

---

# AI Behaviour

When generating interfaces:

Question every element.

Do not add decorations.

Do not chase trends.

Do not imitate Dribbble.

Do not maximize visual complexity.

Optimize for:

Clarity.

Confidence.

Purpose.

Calmness.

---

# Design Language

Interfaces should feel:

Calm.

Confident.

Modern.

Timeless.

Intentional.

Professional.

Elegant without showing off.

Minimal without becoming empty.

Powerful without becoming intimidating.

---

# Visual Inspiration

Take inspiration from systems such as:

Apple

Linear

Stripe

Notion

Arc Browser

Vercel

Figma

But never imitate.

Extract principles.

Not appearances.

---

# Final Rule

If removing an element improves the design...

Remove it.

If adding an element improves the user's understanding...

Add it.

Nothing exists simply because it looks good.

Everything exists because it makes the experience better.
